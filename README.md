# Spring Boot Hexagonal Template

Production-ready Spring Boot template with:

- Hexagonal architecture (ports/adapters)
- Dev, test, and prod profiles
- Liquibase migrations
- H2 for local dev/test and PostgreSQL for prod
- Actuator + Prometheus metrics endpoint
- OpenTelemetry tracing
- Swagger/OpenAPI for API exploration and testing
- JaCoCo + SonarQube config
- Unit and integration tests

## 1. Prerequisites

- JDK 17+
- Maven 3.9+ (or use included Maven Wrapper)
- Git

## 2. Start

Start with the default `dev` profile:

```bash
./mvnw spring-boot:run
```

Windows PowerShell:

```powershell
.\mvnw.cmd spring-boot:run
```

Application base URL:

- `http://localhost:8080`

## 3. API Testing

Once the app is running:

- Swagger UI: `http://localhost:8080/swagger-ui.html`
- OpenAPI JSON: `http://localhost:8080/v3/api-docs`

Resource endpoints currently exposed:

- `POST /api/v1/resources`
- `GET /api/v1/resources`
- `GET /api/v1/resources/{resourceId}`

Example request:

```bash
curl -X POST http://localhost:8080/api/v1/resources \
  -H "Content-Type: application/json" \
  -d '{"name":"Monitor","description":"External display","status":"ACTIVE"}'
```

Example response:

```json
{
  "resourceId": "f78a8762-f661-45c8-b45a-1d154a61f0a5",
  "name": "Monitor",
  "description": "External display",
  "status": "ACTIVE"
}
```

## 4. Resource Model

The domain resource contains:

- `resourceId`: UUID, generated by the application
- `name`: required, max 100 characters
- `description`: optional, max 500 characters
- `status`: optional, defaults to `ACTIVE`, max 50 characters

The application core already includes create, list, get, update, and delete use cases. The public API exposes create, list, and get for now.

## 5. Profiles and Configuration

Configuration uses `.properties` files:

- `src/main/resources/application.properties` (shared defaults)
- `src/main/resources/application-dev.properties`
- `src/main/resources/application-test.properties`
- `src/main/resources/application-prod.properties`

### dev (default)

- H2 in-memory DB
- H2 console: `http://localhost:8080/h2-console`
- Liquibase auto-runs

### test

- Isolated in-memory H2 DB
- Tracing disabled for stable tests

### prod

- PostgreSQL via env vars:
  - `DB_URL`
  - `DB_USERNAME`
  - `DB_PASSWORD`
- H2 console disabled
- Liquibase auto-runs

Run with prod profile:

```bash
./mvnw spring-boot:run -Dspring-boot.run.profiles=prod
```

## 6. Observability

## 8. Architecture

Main package structure:

```text
src/main/java/com/example/demo
|- application
|  |- exception
|  |- port/in
|  |- port/out
|  `- service
|- domain
|  |- model
|  `- validation
`- infrastructure
   `- adapter
      |- in/web
      `- out/persistence
```

Request flow:

1. Inbound adapter (REST controller) receives request.
2. Web mapper converts request DTO to use-case command.
3. Use case in `application.port.in` is invoked.
4. Application service orchestrates domain logic.
5. Outbound port in `application.port.out` is used.
6. Persistence mapper converts domain model to JPA entity and back.

## 9. Error Handling

Global exception handling is centralized in `GlobalExceptionHandler` and returns the company error envelope:

```json
{
  "errors": [
    {
      "code": "VALIDATION_ERROR",
      "message": "name is required",
      "attribute": "name",
      "path": "$.name"
    }
  ]
}
```

Missing resources return the same envelope:

```json
{
  "errors": [
    {
      "code": "RESOURCE_NOT_FOUND",
      "message": "Resource not found: f78a8762-f661-45c8-b45a-1d154a61f0a5",
      "attribute": "resourceId",
      "path": "$.resourceId"
    }
  ]
}
```

## 10. Tests and Quality

Run tests:

```bash
./mvnw test
```

Generate coverage report:

```bash
./mvnw verify
```

JaCoCo report:

- `target/site/jacoco/jacoco.xml`

Sonar config file:

- `sonar-project.properties`

Typical Sonar command:

```bash
./mvnw clean verify sonar:sonar
```

## 11. First Tasks for New Developers

1. Run the app in `dev` and open Swagger UI.
2. Execute `POST /api/v1/resources` and `GET /api/v1/resources`.
3. Run tests (`./mvnw test`).
4. Add PUT/DELETE endpoints when the public API is ready for full CRUD.
